Artifact 01 · Software

BARAD-DUR Tracker

A full-stack, AI-augmented project-management platform I designed and built from the ground up: my most ambitious technical project to date.

My role
Sole designer & developer
Stack
Python · FastAPI · WebSockets · JavaScript
Focus
Full-stack architecture + AI integration
Type
Self-directed project
The Challenge

One place to run the work

In operations, the details that matter, including projects, meetings, tasks, system outages, and the people responsible for each, tend to scatter across spreadsheets, chats, and email. I wanted a single platform that pulled them together, and I wanted to teach myself modern full-stack development and applied AI by building something real rather than following a tutorial.

The result is BARAD-DUR Tracker: an AI-augmented project-management application with dedicated modules for projects, meetings, tasks, outages, and contacts, backed by a Python service layer and a live-updating interface.

What I Built

From empty repository to working platform

A full-stack foundation

A FastAPI/Python service layer exposes the application's data and logic, with a real-time WebSocket system that pushes updates to the interface the moment something changes, with no refresh required.

An AI service layer

I built a custom tool registry and a semantic tool-selection engine so the app can understand natural-language requests, retrieve the right data through embeddings-based search, and let an LLM carry out tasks against real application data.

Five working modules

Projects, meetings, tasks, outages, and contacts, each designed so the information a user needs is grouped together and easy to scan.

An interface people can use

I designed the layouts myself, focusing on clear hierarchy and purposeful structure so the tool feels calm and usable rather than cluttered.

Skills Demonstrated

What this project asked of me

  • Full-stack architecture and API design (FastAPI/Python)
  • Real-time systems with WebSockets
  • Applied AI: tool registries, embeddings retrieval, LLM task execution
  • User-interface and information design
  • Self-directed learning and problem decomposition
  • Hosting, running, and debugging a live application
Reflection

What I learned

Building taught me what tutorials couldn't

The biggest lesson was how much lives between "it works on my machine" and "it works." Standing up a server, wiring real-time updates, and chasing bugs across the stack forced me to think in systems: understanding how each layer depends on the others and isolating a problem methodically instead of guessing. That mindset is the same one I use resolving anomalies in operations, and seeing it transfer gave me real confidence.

AI is a design problem, not just a feature

Integrating an LLM responsibly meant deciding what the model should and shouldn't touch, structuring a tool registry so its actions were predictable, and using embeddings to give it the right context instead of hoping it guessed correctly. I learned that the value isn't the AI itself; it's the careful scaffolding around it that makes it trustworthy and useful.

Scope is a skill

Left unchecked, a personal project can grow forever. I had to make honest calls about what to build now, what to defer, and when a feature was "good enough" to move on. Practicing that judgment, and documenting my decisions so I could pick the work back up later, was as valuable as any line of code.

Why It Matters to My Audience

For an employer, this project is evidence that I can take an idea from a blank repository to a working, AI-enabled application, learning whatever I need along the way. It shows initiative, systems thinking, and the discipline to finish. Those are the qualities I'd bring to an engineering team on day one.
